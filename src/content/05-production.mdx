## 함정 ① — 프록시가 응답을 버퍼링한다

가장 흔하고 가장 짜증나는 함정.

서버는 글자를 한 글자씩 잘 흘려보내는데, 화면에서는 글자가 한 덩어리씩 뚝뚝 끊겨 나오거나 아예 한 번에 펑 하고 나온다. 범인은 보통 **서버와 클라 사이에 끼어있는 nginx, Cloudflare, AWS ALB 같은 프록시**다. "TCP 패킷 효율적으로 쓰자"라는 좋은 의도로 응답을 모아뒀다가 한 번에 내려보낸다.

토글로 직접 체감해보자.

<BufferingDemo />

해결책은 한 줄짜리 헤더 두 개.

<Terminal
  title="서버 응답 헤더에 추가"
  lines={[
    { type: 'output', text: 'Content-Type: text/event-stream' },
    { type: 'output', text: 'Cache-Control: no-cache' },
    { type: 'output', text: 'X-Accel-Buffering: no       # nginx 끄기' },
    { type: 'output', text: 'Connection: keep-alive' },

/>

- **`X-Accel-Buffering: no`**: nginx의 `proxy_buffering` 끄는 시그널
- **`Cache-Control: no-cache`**: 중간 캐시가 응답 가지고 장난 안 치게
- Cloudflare 쓰면 별도로 "Disable Buffering" 룰 추가 필요할 수 있음

<Callout tone="senior">
"이거 안 박으면 dev에선 잘 되다가 stage 들어가면 갑자기 또르륵이 안 나와서 미친다. 우리 작년에 이거 하루 잡고 풀었어."
</Callout>

## 함정 ② — EventSource는 커스텀 헤더를 못 박는다

치명적인 함정. 인증 토큰을 헤더로 보내려고 하면 막힌다.

<Terminal
  title="이게 안 된다"
  lines={[
    { type: 'output', text: '// ❌ 이런 API 없음' },
    { type: 'output', text: 'const es = new EventSource("/api/chat", {' },
    { type: 'output', text: '  headers: { "Authorization": "Bearer abc123" }' },
    { type: 'output', text: '});' },
  ]}
/>

`EventSource` 생성자는 url과 `{ withCredentials: boolean }`만 받는다. 헤더는 못 끼워넣는다. 우회 방법 세 가지.

**방법 A — 쿠키 인증.** `withCredentials: true`로 만들고 서버에서 쿠키로 인증. CORS면 서버에서 `Access-Control-Allow-Credentials: true` 필수.

<Terminal
  title="쿠키 인증"
  lines={[
    { type: 'output', text: 'const es = new EventSource("/api/chat", {' },
    { type: 'output', text: '  withCredentials: true   // 쿠키 같이 보냄' },
    { type: 'output', text: '});' },
  ]}
/>

**방법 B — URL 쿼리에 토큰.** 가장 흔하지만 토큰이 URL에 박혀서 로그/리퍼러로 새기 쉬움. 단명 토큰(짧은 TTL) 권장.

<Terminal
  title="쿼리 파라미터"
  lines={[
    { type: 'output', text: 'const es = new EventSource(`/api/chat?token=${jwt}`);' },
  ]}
/>

**방법 C — `fetch` + `ReadableStream`으로 직접 구현.** EventSource 안 쓰고 fetch로 스트림을 직접 읽으면 헤더 자유. 다만 자동 재연결, 메시지 파싱 다 직접 짜야 함. 또는 `@microsoft/fetch-event-source` 같은 라이브러리 사용.

<Terminal
  title="fetch + ReadableStream 직접 파싱"
  lines={[
    { type: 'output', text: 'const res = await fetch("/api/chat", {' },
    { type: 'output', text: '  headers: { Authorization: `Bearer ${jwt}` }' },
    { type: 'output', text: '});' },
    { type: 'output', text: 'const reader = res.body.getReader();' },
    { type: 'output', text: 'const decoder = new TextDecoder();' },
    { type: 'output', text: 'let buffer = "";' },
    { type: 'output', text: 'while (true) {' },
    { type: 'output', text: '  const { value, done } = await reader.read();' },
    { type: 'output', text: '  if (done) break;' },
    { type: 'output', text: '  buffer += decoder.decode(value);' },
    { type: 'output', text: '  // buffer에서 \\n\\n 찾아서 메시지 잘라내기...' },
    { type: 'output', text: '}' },
    { type: 'comment', text: '자동 재연결, 메시지 파싱 다 직접 짜야 함' },
  ]}
/>

<Callout tone="senior">
"방법 C가 코드는 많아 보이는데 운영 가면 결국 이걸로 수렴해. 헤더도 자유롭고, abort도 깔끔하고, 진행 중에 강제 종료(`Stop` 버튼)도 잘 동작하거든. EventSource는 데모용으로는 좋지만 챗봇 같은 실전 LLM 앱에선 `fetch` 기반이 사실상 표준이야."
</Callout>

## 함정 ③ — HTTP/1.1 동시 연결 6개 제한

브라우저는 한 origin당 동시 HTTP/1.1 연결을 6개까지만 연다. SSE 한 개가 연결을 계속 점유하니까, **탭 6개 열고 SSE 받으면 7번째 탭은 멈춰버린다**.

<Callout tone="danger">
한 탭에서 SSE 여러 개를 동시에 여는 패턴도 마찬가지. 같은 도메인 SSE 5~6개만 동시에 열어도 같은 도메인의 다른 API 호출이 다 막힐 수 있다.
</Callout>

해결책: **HTTP/2 또는 HTTP/3으로 운영하기**. HTTP/2부터는 한 TCP 연결 위에 여러 스트림이 멀티플렉싱돼서 6개 제한이 사실상 사라진다. 요즘 GitHub Pages, Vercel, Cloudflare 등 대부분의 호스팅은 기본 HTTP/2 이상.

<Terminal
  title="서버 HTTP/2 확인"
  lines={[
    { type: 'output', text: 'curl -I --http2 https://api.yourservice.com/' },
    { type: 'output', text: 'HTTP/2 200' },
    { type: 'comment', text: 'HTTP/2가 나오면 OK. HTTP/1.1만 나오면 인프라 점검' },
  ]}
/>

## 함정 ④ — CORS와 `withCredentials`

다른 도메인 SSE 받으려면 CORS 세팅 필수. 평범한 GET이라 일반적인 CORS 룰이 적용되지만, **쿠키 인증을 쓰면 한 가지 더 필요**.

<Terminal
  title="서버 응답 헤더"
  lines={[
    { type: 'output', text: 'Access-Control-Allow-Origin: https://app.example.com' },
    { type: 'output', text: 'Access-Control-Allow-Credentials: true' },
    { type: 'comment', text: 'Allow-Origin 에 와일드카드(*) 쓰면 Credentials 안 됨 — 명시 도메인이어야 함' },
  ]}
/>

<Callout tone="danger">
`Access-Control-Allow-Origin: *` 과 `Access-Control-Allow-Credentials: true` 를 같이 쓰면 브라우저가 거부한다. 쿠키 인증을 쓸 거면 origin을 명시적으로 박아야 한다.
</Callout>

## 함정 ⑤ — 좀비 연결과 메모리 누수

가장 보이지 않는 함정. 사용자가 페이지를 떠났는데도 서버 입장에선 연결이 살아있는 것처럼 보이는 경우가 있다 (특히 모바일에서 백그라운드로 가면). 챗봇 서버가 토큰을 계속 만들어서 닫힌 소켓에 쓰다가, 어느 순간 알아채고 끊어진다.

방어 두 가지.

1. **서버에서 주기적으로 keep-alive comment 보내기** — `: keep-alive\n\n` 을 15~30초마다. 보내다 실패하면 그때 클라가 죽었다는 걸 빨리 알아챔.
2. **클라에서 `beforeunload` 또는 `visibilitychange` 에 `es.close()`** — 사용자가 떠날 때 명시적으로 닫기.

<Terminal
  title="클라 정리 코드"
  lines={[
    { type: 'output', text: 'window.addEventListener("beforeunload", () => es.close());' },
    { type: 'output', text: '' },
    { type: 'output', text: '// 또는 LLM 응답 도중 사용자가 "중단" 누를 때' },
    { type: 'output', text: 'stopBtn.onclick = () => {' },
    { type: 'output', text: '  es.close();' },
    { type: 'output', text: '  // 백엔드도 같이 끊고 싶으면 별도 DELETE 호출 같은 거 필요' },
    { type: 'output', text: '};' },
  ]}
/>

<Callout tone="senior">
"마지막 거 진짜 중요해. 우리 LLM API 비싸잖아? 사용자가 '중단' 눌렀는데 서버가 그걸 몰라서 토큰 1000개 더 만들어버리면 비용 그대로 청구돼. abort 시그널 잘 전파시키는 게 운영 비용 직결."
</Callout>

## 회고 — 5에피소드의 핵심을 한 줄씩

| 화 | 머리에 박을 한 줄 |
|---|---|
| Ep.1 | HTTP 응답은 끝낼 의무가 없다. SSE는 그 사실의 표준화일 뿐 |
| Ep.2 | EventSource는 빈 줄(`\n\n`)을 만날 때마다 모아둔 라인을 한 메시지로 발사한다 |
| Ep.3 | `data:` 외에 `event:`, `id:`, `retry:` 세 라인이 더 있고 — 각자 깔끔한 용도가 있다 |
| Ep.4 | 자동 재연결 + `Last-Event-ID` 헤더가 끊김을 사실상 안 보이게 만든다 |
| Ep.5 | 운영의 진짜 함정은 SSE 자체가 아니라 그 주변 인프라 (프록시, 인증, HTTP 버전) |

## 마무리

5에피소드 동안 냥사원과 같이 SSE의 본질부터 실전 함정까지 한 바퀴 돌았다. 이제 응답이 안 흘러나오는 챗봇을 보면 어디부터 까볼지 머릿속 지도가 있을 거다.

마지막으로 한 줄. **SSE는 새 마법이 아니다 — HTTP 응답을 안 닫는다는 단순한 발상이 정밀하게 표준화된 것**이다. 그 단순함이 자동 재연결, 자동 메시지 파싱, 자동 책갈피까지 공짜로 끼워주는 비결이다.

이제 진짜 또르륵 챗봇을 만들 차례다.
