## 점심 먹고 자리로 돌아온 냥사원

오전에 부장님 화이트보드 강의를 듣고 머리가 시원해진 냥사원. 점심으로 회사 근처 김치찌개를 먹고 자리로 돌아왔다. 이제 진짜로 코드를 짜야 한다.

서버 쪽은 그렇다 치자 — 응답을 안 닫은 채로 `data: ...\n\n` 포맷을 한 줄씩 흘려보내면 된다. 그런데 **클라이언트는?** 그 흘러오는 바이트를 도대체 어떻게 받아서 글자로 조립하지?

<Callout tone="story">
"손으로 `fetch`로 받아서 `\n\n` 위치 찾고 잘라야 하나? 자르고 나서는 `data: ` 접두사도 떼야 하고… 이거 생각보다 빡센데?"
</Callout>

냥사원이 한숨을 쉬는데 옆자리 냥차장이 슬쩍 모니터를 본다.

<Callout tone="senior">
"어, 그거 브라우저에 이미 있어. `EventSource`라고. `new EventSource(url)` 한 줄이면 끝나."
</Callout>

## 클라이언트 한 줄의 정체

냥차장 말이 사실이라면, 진짜 코드는 이렇게 짧다.

<Terminal
  title="chat-client.js"
  lines={[
    { type: 'output', text: 'const es = new EventSource("/api/chat");' },
    { type: 'output', text: '' },
    { type: 'output', text: 'es.onmessage = (e) => {' },
    { type: 'output', text: '  console.log("받음:", e.data);' },
    { type: 'output', text: '  appendToChat(e.data);' },
    { type: 'output', text: '};' },
    { type: 'output', text: '' },
    { type: 'output', text: 'es.onerror = (e) => {' },
    { type: 'output', text: '  console.warn("연결 문제, 알아서 재시도 중", e);' },
    { type: 'output', text: '};' },
  ]}
/>

이게 끝이다. `new WebSocket(...)` 쓸 때처럼 `onopen`/`onmessage`/`onclose`/`onerror`를 일일이 다 다룰 필요도 없고, 메시지 프레임을 직접 파싱할 필요도 없다.

그럼 이 한 줄짜리 마법은 **속에서 무슨 일을 하고 있는 걸까**? 까보자.

<Callout tone="success" title="이 화의 핵심">
`EventSource`는 (1) `text/event-stream`을 받는 GET 요청을 보내고, (2) 응답 바이트를 누적해서 빈 줄(`\n\n`)을 만날 때마다 그때까지의 라인들을 모아 **하나의 메시지로 파싱**해서 `message` 이벤트로 발사한다. 우리가 직접 안 해도 되는 건 이 파싱 단계 전부다.
</Callout>

## 직접 만져보기 — 바이트가 메시지가 되는 순간

말로만 들으면 또 안 와닿는다. 아래에서 서버가 한 글자씩 흘려보내는 걸 직접 보자. 가운데 박스(누적 버퍼)에 글자가 쌓이다가 **빈 줄(`\n\n`)을 만나는 순간** 오른쪽에 메시지가 발사된다.

샘플 페이로드를 바꿔보거나, **"한 글자만 ▷"** 버튼으로 한 글자씩 직접 보내보면 파싱이 어디서 일어나는지 또렷이 보인다.

<ByteParser />

<Callout tone="info" title="관찰 포인트 1: 빈 줄이 메시지를 만든다">
"정상 메시지 1개" 샘플로 한 글자씩 보내보자. `data: 안녕하세요!` 까지 입력해도 오른쪽엔 아무것도 안 뜬다. `\n`을 한 번 더 친 순간(= 빈 줄을 만든 순간) 비로소 메시지가 발사된다.
</Callout>

<Callout tone="danger" title="관찰 포인트 2: 빈 줄 없으면 영영 안 옴">
"⚠️ 빈 줄 없음" 샘플을 보내보자. 서버가 데이터를 다 보냈는데도 EventSource는 메시지를 안 발사한다. 메시지 경계를 못 찾았기 때문이다. **실전에서 가장 흔한 실수** — 서버에서 `\n\n` 안 쓰고 `\n` 하나만 쓰는 거.
</Callout>

<Callout tone="info" title="관찰 포인트 3: data 여러 줄은 \n으로 합쳐진다">
"data 여러 줄" 샘플을 보내보자. 서버가 `data:` 라인을 3번 쓰면, EventSource가 그걸 `\n`으로 join해서 한 개의 message로 만든다. 멀티라인 응답 보낼 때 쓰는 패턴.
</Callout>

## "그럼 WebSocket이랑 비교했을 때 무슨 코드가 줄어든 거예요?"

오후 회의 마치고 돌아온 냥차장이 한 줄 비교를 보여준다.

<Terminal
  title="동일한 챗봇을 WebSocket으로 짜면..."
  lines={[
    { type: 'output', text: 'const ws = new WebSocket("wss://api/chat");' },
    { type: 'output', text: '' },
    { type: 'output', text: 'ws.onopen = () => ws.send(JSON.stringify({type:"start"}));' },
    { type: 'output', text: 'ws.onmessage = (e) => {' },
    { type: 'output', text: '  // 1) 메시지 포맷 직접 정해야 함 (JSON? binary?)' },
    { type: 'output', text: '  const frame = JSON.parse(e.data);' },
    { type: 'output', text: '  if (frame.type === "token") appendToChat(frame.text);' },
    { type: 'output', text: '};' },
    { type: 'output', text: 'ws.onclose = () => {' },
    { type: 'output', text: '  // 2) 재연결 직접 구현해야 함' },
    { type: 'output', text: '  setTimeout(() => reconnect(), 3000);' },
    { type: 'output', text: '};' },
    { type: 'output', text: 'ws.onerror = (e) => { /* ... */ };' },
    { type: 'comment', text: 'EventSource는 (1)(2) 둘 다 안 해도 됨' },
  ]}
/>

요점: SSE가 WebSocket보다 "기능이 적어서 짧다"가 아니라, **단방향이라는 제약을 받아들이는 대신 표준화된 포맷 + 자동 재연결을 공짜로 받는** 거다.

<Callout tone="aha">
"부장님이 'WebSocket 튜토리얼 그만 봐'라고 하신 이유를 이제 알 것 같아요. 챗봇 응답 스트리밍은 클라가 서버에 자주 뭘 보낼 일이 없는데, 괜히 양방향 도구를 들고 와서 onmessage 파서랑 재연결 로직을 직접 짜고 있었던 거네요."
</Callout>

## 한 가지 더 — JSON 토큰을 흘려보낼 때

실전에선 `data:` 안에 그냥 글자가 아니라 **JSON**을 넣는 경우가 많다. OpenAI도 그렇게 한다.

<Terminal
  title="실제 LLM API의 SSE 응답 예시"
  lines={[
    { type: 'output', text: 'data: {"token":"안","finish":false}' },
    { type: 'output', text: '' },
    { type: 'output', text: 'data: {"token":"녕","finish":false}' },
    { type: 'output', text: '' },
    { type: 'output', text: 'data: {"token":"하세요","finish":true}' },
    { type: 'output', text: '' },
    { type: 'output', text: 'data: [DONE]' },
    { type: 'output', text: '' },
  ]}
/>

이러면 클라이언트는 이렇게 받는다.

<Terminal
  title="JSON 파싱 추가"
  lines={[
    { type: 'output', text: 'es.onmessage = (e) => {' },
    { type: 'output', text: '  if (e.data === "[DONE]") { es.close(); return; }' },
    { type: 'output', text: '  const { token, finish } = JSON.parse(e.data);' },
    { type: 'output', text: '  appendToChat(token);' },
    { type: 'output', text: '  if (finish) markEndOfTurn();' },
    { type: 'output', text: '};' },
  ]}
/>

핵심은 변하지 않는다. EventSource는 여전히 빈 줄을 만날 때마다 `e.data`에 텍스트를 채워서 발사할 뿐이다. **그 텍스트가 우연히 JSON이라서 우리가 한 번 더 파싱하는 것**일 뿐.
