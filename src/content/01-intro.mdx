## 또르를 소개합니다

또르는 1인 챗봇 개발자다. 야밤에 라면 먹으면서 사이드 프로젝트로 챗봇을 하나 만들었다. 자기 입으로 "또르륵 챗봇"이라고 부르는데, 이유는 단순하다 — 친구 **지피티**처럼 글자가 또르륵 또르륵 흘러나오게 만들고 싶었기 때문이다.

그런데 막상 돌려보니 또르의 챗봇은 답을 한참 끓이다가 한 번에 떡 뱉는다. 똑같은 LLM API를 갖다 썼는데 왜 지피티는 글자가 흘러나오고 또르 거는 한방에 나오는 거지?

<Callout tone="story" title="또르의 한탄">
"내 챗봇은 답을 만드는 동안 사용자를 멍 때리게 만든다. 5초간 흰 화면이 떠 있다가 답이 펑 하고 뜬다. 지피티는 같은 5초 동안 글자가 또르륵 흘러나와서 마법처럼 보인다. 같은 LLM인데 도대체 뭐가 다른 거야?"
</Callout>

## 또르의 첫 번째 가설: "HTTP가 원래 그런 거 아니야?"

또르는 학교에서 배운 HTTP를 떠올린다.

> "HTTP는 요청 한 번에 응답 한 번. 응답은 다 만들어서 한 번에 보내는 거니까, 글자가 줄줄 나오게 하려면 다른 프로토콜이 필요한 거 아닐까?"

그럴듯하지만 **틀린 모델**이다. 이 오해 하나가 또르를 한 시간 동안 WebSocket 튜토리얼만 뒤지게 만들었다.

진실은 이렇다. HTTP 응답은 **"끝낼 의무가 없다"**. 서버가 응답을 안 닫으면 클라이언트는 계속 받는다. 그게 다다.

<Callout tone="success" title="이 화의 핵심">
HTTP 응답은 "한 번에 보내고 끝"이 아니라 "열어둔 TCP 연결 위에 바이트를 흘려보내는 것"이다. 서버가 응답을 일부러 안 닫으면, 그 위로 새 글자를 계속 보낼 수 있다. **SSE는 이걸 표준화한 약속일 뿐이다.**
</Callout>

## 직접 만져보기 — 한 번에 보내기 vs 흘려보내기

말로만 들으면 안 와닿는다. 둘을 동시에 호출해서 직접 비교해보자. 똑같은 답을 만들지만, 왼쪽 또르 챗봇은 다 만들고 한 번에 보내고, 오른쪽 지피티는 글자가 생기는 족족 흘려보낸다.

<ResponseRace />

<Callout tone="info" title="관찰 포인트">
두 챗봇이 답을 만드는 시간은 똑같다. 차이는 **"언제 사용자에게 보여주냐"**다. 왼쪽은 답이 다 만들어질 때까지 한 글자도 안 보여주고, 오른쪽은 첫 글자가 만들어지자마자 바로 보여준다. 네트워크 위에서 흐른 바이트(아래 회색 박스)를 비교해 보면 차이가 더 또렷하다 — 왼쪽은 한 번에 큰 덩어리, 오른쪽은 잘게 쪼개진 `data: …` 라인이 계속 추가된다.
</Callout>

## "그럼 SSE가 새 프로토콜이라는 거야?"

아니다. **그냥 HTTP다.** SSE = "Server-Sent Events"의 약자. 풀어쓰면 "서버가 알아서 보내주는 이벤트들". 실체는 다음 두 가지 약속뿐이다.

1. 서버는 응답 헤더에 `Content-Type: text/event-stream`을 박는다.
2. 그리고 응답을 닫지 않은 채로, 정해진 텍스트 포맷(`data: ...\n\n`)을 한 줄씩 흘려보낸다.

이게 끝이다. 새로운 포트도, 새로운 핸드셰이크도, 새로운 라이브러리도 필요 없다. **그냥 HTTP GET 요청 하나에 응답을 안 닫는 것**뿐이다.

<Callout tone="aha" title="또르의 깨달음">
"엥? 그럼 SSE는 새 프로토콜도 아니고 그냥 HTTP를 안 닫는 거잖아? 그동안 WebSocket 튜토리얼만 뒤지고 있었네…"
</Callout>

## 그럼 WebSocket이랑 뭐가 다른데?

또르의 머릿속 두 번째 가설("WebSocket 써야 하나?")도 짚고 가자. 둘 다 "실시간"이라는 단어로 묶여있지만 결이 다르다.

| | **WebSocket** | **SSE** |
|---|---|---|
| 방향 | 양방향 (서버↔클라) | **단방향** (서버→클라만) |
| 프로토콜 | `ws://`, `wss://` (HTTP에서 업그레이드) | 그냥 HTTP |
| 메시지 포맷 | 직접 정해야 함 (JSON? binary?) | **정해져 있음** (`data: ...\n\n`) |
| 자동 재연결 | 직접 구현 | **브라우저가 알아서** |
| 인증 헤더 | 자유롭게 | 못 박는다 (다음 화의 함정) |
| 클라이언트 코드 | `new WebSocket(...)` + 이벤트 핸들러 잔뜩 | `new EventSource(url)` 한 줄 |

규칙 하나만 기억하자. **서버 → 클라 단방향이면 SSE가 압도적으로 단순하다.** 채팅 응답 토큰 스트리밍, 알림, 진행률, 주식 가격 푸시 — 다 단방향이다. 클라가 서버에 뭘 보낼 필요가 있을 때만(채팅방의 양방향 대화, 게임) WebSocket이 필요하다.

## 잠깐, 그럼 부작용은 없어?

좋은 질문이다. 응답을 안 닫는다는 건 곧 **TCP 연결을 계속 점유한다**는 뜻이다. 여기서 두 가지 주의가 따라온다.

- **함정 1.** 옛날 브라우저는 한 도메인에 동시 HTTP/1.1 연결을 6개까지만 열 수 있다. 탭을 6개 열고 SSE를 6개 동시에 받으면 7번째 탭은 멈춰버린다. **해결: HTTP/2를 쓰면 한 연결 위에 여러 스트림이 멀티플렉싱돼서 문제없다.** (5화에서 자세히)
- **함정 2.** 서버와 클라 사이에 끼어 있는 nginx, Cloudflare 같은 프록시가 응답을 **버퍼링**해버리면, 또르륵이 또~륵이 되거나 아예 한 번에 펑 하고 나온다. (이것도 5화에서 회수)

지금은 "SSE 자체는 새 마법이 아니다, 응답을 안 닫는 HTTP일 뿐이다"만 머리에 단단히 박으면 된다.
