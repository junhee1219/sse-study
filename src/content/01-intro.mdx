## 냥테크의 어느 월요일 아침

냥사원은 입사한 지 한 달째다. 사이드 프로젝트로 만든 챗봇 데모를 오늘 부장님께 보여드리기로 했다. 발표 자료까지 깔끔하게 준비해놨다.

화면을 켜고 자신 있게 "안녕!" 입력. 화면이 5초간 멈추더니, 펑 하고 답이 통째로 떠오른다.

냥부장이 한 발 떨어져서 보다가 한 마디.

<Callout tone="boss">
"...근데 너 왜 글자 또르륵 안 나오게 했냐? 옆 회사 챗봇은 또르륵 나오던데?"
</Callout>

냥사원의 등에 식은땀.

<Callout tone="story">
"네? HTTP는 응답을 한 번에 보내는 거 아닌가요? 글자가 또르륵 나오게 하려면 WebSocket을 써야 하는 줄 알았는데요…"
</Callout>

냥부장이 의자를 끌어와 앉는다. 손에는 따뜻한 그란데 라떼 한 잔.

<Callout tone="boss">
"내가 5분만 까볼게. 너 HTTP 응답은 끝낼 의무가 없다는 거 알아?"
</Callout>

## 냥부장의 화이트보드 — HTTP 응답은 끝낼 의무가 없다

학교에서 배운 HTTP는 보통 이렇다. "요청 한 번, 응답 한 번. 응답은 다 만들어서 한 덩어리로 보내고 끝."

그럴듯하지만 **틀린 모델**이다. 이 오해 하나가 냥사원을 한 시간 동안 WebSocket 튜토리얼만 뒤지게 만들었다.

진실은 이렇다. HTTP 응답은 **"끝낼 의무가 없다"**. 서버가 응답을 안 닫으면 클라이언트는 계속 받는다. 그게 전부다.

<Callout tone="success" title="이 화의 핵심">
HTTP 응답은 "한 번에 보내고 끝"이 아니라 "열어둔 TCP 연결 위에 바이트를 흘려보내는 것"이다. 서버가 응답을 일부러 안 닫으면, 그 위로 새 글자를 계속 보낼 수 있다. **SSE는 이걸 표준화한 약속일 뿐이다.**
</Callout>

## 직접 만져보기 — 한 번에 보내기 vs 흘려보내기

말로만 들으면 안 와닿는다. 두 챗봇에게 같은 질문을 동시에 던져서 직접 비교해보자. 왼쪽은 냥사원이 짠 챗봇(다 만들고 한 번에 보냄), 오른쪽은 옆 회사 챗봇(생긴 만큼 즉시 흘려보냄). 답을 만드는 시간은 **똑같다**.

<ResponseRace />

<Callout tone="info" title="관찰 포인트">
두 챗봇이 답을 만드는 시간은 똑같다. 차이는 **"언제 사용자에게 보여주냐"**다. 왼쪽은 답이 다 만들어질 때까지 한 글자도 안 보여주고, 오른쪽은 첫 글자가 만들어지자마자 바로 보여준다. 네트워크 위에서 흐른 바이트(아래 회색 박스)를 비교해 보면 차이가 더 또렷하다 — 왼쪽은 한 번에 큰 덩어리, 오른쪽은 잘게 쪼개진 `data: …` 라인이 계속 추가된다.
</Callout>

## "그럼 SSE가 새 프로토콜이라는 거예요?"

냥사원이 조심스럽게 묻는다. 냥부장이 마우스를 가리키며 답한다.

<Callout tone="boss">
"아니, 그냥 HTTP야. SSE는 약속 두 가지 합쳐놓은 거야. 첫째, 서버는 응답 헤더에 `Content-Type: text/event-stream` 박는다. 둘째, 응답을 안 닫은 채로 `data: ...\\n\\n` 포맷을 한 줄씩 흘려보낸다. 끝."
</Callout>

새 포트도, 새 핸드셰이크도, 새 라이브러리도 필요 없다. **그냥 HTTP GET 요청 하나에 응답을 안 닫는 것**뿐이다. 풀어쓰면 "Server-Sent Events" — 서버가 알아서 보내주는 이벤트들.

<Callout tone="aha">
"엥? 그럼 SSE는 새 프로토콜도 아니고 그냥 HTTP를 안 닫는 거잖아요? 어제 밤새 WebSocket 튜토리얼만 뒤지고 있었는데…"
</Callout>

## 그럼 WebSocket이랑 뭐가 다른데?

냥사원의 머릿속 두 번째 가설("WebSocket 써야 하나?")도 짚고 가자. 둘 다 "실시간"이라는 단어로 묶여 있지만 결이 다르다.

| | **WebSocket** | **SSE** |
|---|---|---|
| 방향 | 양방향 (서버↔클라) | **단방향** (서버→클라만) |
| 프로토콜 | `ws://`, `wss://` (HTTP에서 업그레이드) | 그냥 HTTP |
| 메시지 포맷 | 직접 정해야 함 (JSON? binary?) | **정해져 있음** (`data: ...\n\n`) |
| 자동 재연결 | 직접 구현 | **브라우저가 알아서** |
| 인증 헤더 | 자유롭게 | 못 박는다 (5화의 함정) |
| 클라이언트 코드 | `new WebSocket(...)` + 이벤트 핸들러 잔뜩 | `new EventSource(url)` 한 줄 |

규칙 하나만 기억하자. **서버 → 클라 단방향이면 SSE가 압도적으로 단순하다.** 챗봇 토큰 스트리밍, 알림, 진행률, 주식 가격 푸시 — 다 단방향이다. 클라가 서버에 자주 뭘 보낼 때만(채팅방의 양방향 대화, 게임) WebSocket이 필요하다.

## 실제로는 어떻게 생겼나

말 들으니 궁금하다. 옆 회사 챗봇 서버 응답을 `curl`로 까보면 이렇게 생겼다.

## 잠깐, 그럼 부작용은 없어?

좋은 질문이다. 응답을 안 닫는다는 건 곧 **TCP 연결을 계속 점유한다**는 뜻이다. 여기서 두 가지 함정이 따라온다.

<Callout tone="senior">
"브라우저는 한 도메인에 동시 HTTP/1.1 연결을 6개까지만 열어. 탭 6개 열고 SSE 6개 동시에 받으면 7번째 탭은 멈춰버려. 우리 작년에 그것 때문에 한 시간 날렸어. **해결책은 HTTP/2 쓰는 거** — 한 연결 위에 여러 스트림이 멀티플렉싱되니까 문제없어. 5화에서 자세히 볼게."
</Callout>

<Callout tone="senior">
"그리고 서버와 클라 사이에 끼어있는 nginx, Cloudflare 같은 프록시가 응답을 **버퍼링**해버리면, 또르륵이 또~륵 되거나 아예 한 번에 펑 하고 나와. 이것도 5화에서 회수할게."
</Callout>

지금은 "SSE 자체는 새 마법이 아니다, 응답을 안 닫는 HTTP일 뿐이다"만 머리에 단단히 박으면 된다.
